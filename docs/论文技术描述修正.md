## 1. 异常处理机制

### 实际实现

项目通过 `ErrorCode` 枚举定义错误码体系：

```java
public enum ErrorCode {
    SUCCESS(0, "ok"),
    PARAMS_ERROR(40000, "请求参数错误"),
    NOT_LOGIN_ERROR(40100, "未登录"),
    NO_AUTH_ERROR(40101, "无权限"),
    NOT_FOUND_ERROR(40400, "请求数据不存在"),
    FORBIDDEN_ERROR(40300, "禁止访问"),
    SYSTEM_ERROR(50000, "系统内部异常"),
    OPERATION_ERROR(50001, "操作失败");
}
```

自定义 `BusinessException` 继承 `RuntimeException`，封装错误码与消息。全局异常处理器 `GlobalExceptionHandler` 使用 `@RestControllerAdvice` 分别捕获 `BusinessException` 和 `RuntimeException`。

### 论文描述建议

> 项目通过 `ErrorCode` 枚举定义涵盖参数校验、认证授权、资源不存在及系统级异常的错误码体系，每个错误码包含五位数字标识和描述性消息。自定义 `BusinessException` 封装错误码与消息，通过 `GlobalExceptionHandler` 全局异常处理器统一捕获并返回 `BaseResponse` 响应体。`BusinessException` 返回对应错误码和消息，未预期的 `RuntimeException` 返回系统错误码（50000）并隐藏具体错误细节，仅记录日志用于问题排查。

## 2. 响应封装

### 实际实现

```java
@Data
public class BaseResponse<T> implements Serializable {
    private int code;
    private T data;
    private String message;
}
```

通过 `ResultUtils` 工具类统一构造响应，成功时 code=0，失败时填充对应错误码。`JsonConfig` 配置类全局注册 Jackson 序列化器解决 Long 类型精度丢失问题。

### 论文描述建议

> 所有接口返回统一的 `BaseResponse<T>` 结构，包含 `code` 状态码、`message` 提示、`data` 业务数据。Controller 通过 `ResultUtils` 工具类构造响应，成功时 code=0，失败时填充对应错误码。
>
> JavaScript 的 Number 类型安全整数范围仅到 $2^{53}-1$，而 Java Long 类型可达 $2^{63}-1$。本项目通过 `Jackson2ObjectMapperBuilderCustomizer` 全局注册自定义 Jackson 模块：序列化时将 Long 转为字符串，反序列化时同时接受数字和字符串两种格式，实现双向兼容。相比逐字段添加注解，全局配置只需维护一处，且能覆盖所有 Long 字段（包括嵌套对象中的字段）。

## 3. AI 图片生成（Gemini SDK 封装）

### 实际实现

- `AiPictureGeneratorServiceImpl` 核心服务类
- `Gemini` 组件通过 `@PostConstruct` 初始化 SDK 客户端
- 支持纯文本或多模态（文本+最多14张图片）输入
- 同步调用 Gemini API，非异步队列
- 聊天历史通过 `chat_history` 和 `chat_history_picture` 表持久化

### 论文描述建议

> AI 图片生成模块封装了 Google Gemini SDK，核心服务类 `AiPictureGeneratorServiceImpl` 负责处理生图请求。SDK 客户端通过 `@PostConstruct` 生命周期钩子完成初始化，从 Spring 配置文件注入 `apiKey`、`baseUrl` 和 `modelName`，其中 `baseUrl` 支持配置自定义代理地址。
>
> 生成接口支持两种输入模式：纯文本描述，或文本描述配合最多 14 张参考图片。当请求包含参考图片时，系统首先校验图片归属权限（非管理员只能使用私有空间的图片），然后将图片从 COS 下载为字节流，优先使用缩略图版本以减少网络传输。文本 prompt 与图片字节流共同组装为 Gemini API 所需的 `Content` 对象。
>
> Gemini API 调用采用同步阻塞模式。`GenerateContentConfig` 配置了 `responseModalities` 为 `IMAGE` 和 `TEXT`，同时接受图片和文本两种响应类型，并包含系统指令定义 AI 的角色为"高级艺术总监"。
>
> 响应解析通过 `inlineData` 和 `text` 字段区分返回类型。若存在 `inlineData`，提取图片数据上传至 COS；若存在 `text`，提取文本内容。两类响应均记录至聊天历史表。`ChatHistoryPicture` 关联表记录每次对话的输入输出图片，`pictureType` 字段区分 `INPUT` 和 `OUTPUT`，支持用户追溯历史生成记录的完整上下文。

## 4. 扩图功能（阿里云 DashScope）

### 实际实现

- 两个 REST 接口：`create_task` 创建任务，`get_task` 查询状态
- 通过 `X-DashScope-Async: enable` header 启用异步模式
- 前端负责轮询，后端无定时任务
- 前端获取结果后通过 `UrlPictureUpload` 存储到 COS

### 论文描述建议

> 扩图功能基于阿里云 DashScope 的 `image-out-painting` 模型实现。由于 AI 扩图处理耗时较长，API 采用异步调用模式：创建任务时在 HTTP 请求头设置 `X-DashScope-Async: enable`，阿里云返回任务 ID 而非同步等待结果。
>
> 后端封装了两个 REST 接口。`create_task` 负责提交扩图请求，校验用户权限后将原图 URL 和扩图参数（如 xScale、yScale 扩展比例，angle 旋转角度，outputRatio 输出宽高比）发送至阿里云 API。`get_task` 负责查询任务状态，直接透传阿里云的响应，状态包括 `PENDING`（排队中）、`RUNNING`（处理中）、`SUCCEEDED`（成功）、`FAILED`（失败）。
>
> 轮询逻辑由前端控制。前端获取 taskId 后定时调用查询接口，直到状态变为成功或失败。成功响应中包含扩图后的临时图片 URL，前端随后调用后端的 URL 上传接口，将图片从阿里云拉取并存储至腾讯云 COS，同时触发图片压缩和缩略图生成。这种设计将轮询压力分散到各客户端，后端保持无状态，便于水平扩展。

## 5. 分布式会话管理

### 实际实现

- 使用 Spring Session + Redis，**非 JWT**
- 标准 Servlet API：`request.getSession().setAttribute/getAttribute`
- Session 超时 30 天
- AOP 切面 `AuthInterceptor` 实现 `@AuthCheck` 权限校验

### 论文描述建议

> 在分布式系统中，基于服务器本地内存的会话管理在集群环境下面临会话状态难以同步的问题。本项目采用 **Spring Session + Redis** 实现分布式会话管理，将 HttpSession 数据集中存储于 Redis，集群内各节点共享同一份会话数据。
>
> 具体实现上，项目引入 `spring-session-data-redis` 依赖，配置 Redis 连接参数和 Session 超时时间（30 天）。代码层面保持使用标准 Servlet API：登录时通过 `request.getSession().setAttribute()` 将用户信息存入 Session；后续请求通过 `request.getSession().getAttribute()` 获取登录状态。Spring Session 框架自动拦截 Session 读写操作，将数据序列化存储至 Redis，并在请求到达时从 Redis 恢复 Session。
>
> 权限控制通过自定义 AOP 切面实现。`AuthInterceptor` 类使用 `@Around` 注解拦截带有 `@AuthCheck` 注解的方法，从 Session 中获取当前登录用户，校验其角色是否满足接口要求的权限等级。若校验失败，抛出 `NO_AUTH_ERROR` 异常由全局异常处理器统一响应。
>
> 该方案的优势在于：代码无需关心分布式细节，继续使用标准的 HttpSession API；Spring Session 框架透明地完成 Session 外部化存储，天然支持集群部署。Redis 的高性能读写保障了会话访问效率，Session 过期机制由 Redis 和 Spring Session 共同维护。

## 6. 权限校验机制

### 原文与实际对比

| 原文描述 | 实际情况 |
|---------|---------|
| RBAC 三层模型（用户-角色-权限） | ❌ 只有简单角色字符串比较 |
| 权限标识符（如 "image:delete"） | ❌ 没有权限标识，只有角色名 |
| `@RequiresPermission` 注解 | ❌ 实际是 `@AuthCheck(mustRole)` |
| 角色表 + 权限表 + 中间表 | ❌ 角色直接存在用户表 userRole 字段 |
| "三步流程：提取角色、查询权限集合、校验" | ❌ 只有一步：比较字符串 |
| Redis 发布订阅权限更新 | ❌ 没有 |
| 权限缓存 + @CacheEvict | ❌ 没有 |
| "500毫秒延迟" | ❌ 虚构数据 |
| 角色继承树 | ❌ 没有继承关系 |

### 实际实现

#### 数据模型

用户表中直接存储角色和等级字段，而非通过关联表：

```sql
-- user 表关键字段
user_role   VARCHAR(20)   -- "admin" | "user" | "vip"
vip_level   VARCHAR(20)   -- "standard" | "pro" | "max"
```

#### 角色与等级枚举

```java
// 用户角色枚举
public enum UserRoleEnum {
    USER("普通用户", "user"),
    ADMIN("管理员", "admin"),
    VIP("VIP用户", "vip");
}

// VIP 等级枚举
public enum UserVIPLevelEnum {
    STANDARD("标准版", "standard"),
    PRO("专业版", "pro"),
    MAX("旗舰版", "max");
}
```

#### 自定义注解

```java
// 角色校验注解
@Target({ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface AuthCheck {
    String mustRole() default "";
}

// VIP 等级校验注解
@Target({ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface VipLevelCheck {
    String mustLevel() default "";
}
```

#### AOP 切面实现

```
@AuthCheck(mustRole = "admin")
       ↓
AuthInterceptor (@Around)
       ↓
RequestContextHolder.getRequestAttributes()
       ↓
userService.getLoginUser(request)
       ↓
比较 user.getRole().equals(mustRole)
       ↓
相等 → joinPoint.proceed()
不等 → throw BusinessException(NO_AUTH_ERROR)
```

#### 校验流程

**角色校验（AuthInterceptor）：**
1. 从 `RequestContextHolder` 获取当前请求
2. 调用 `userService.getLoginUser()` 从 Session 获取登录用户
3. 将注解中的 `mustRole` 转换为 `UserRoleEnum`
4. 比较用户的 `userRole` 与要求的角色
5. 管理员角色直接放行所有接口

**VIP 等级校验（VipLevelInterceptor）：**
1. 获取登录用户及其 `vipLevel` 字段
2. 管理员角色跳过 VIP 等级校验
3. 根据要求的等级进行层级比较：
   - 要求 STANDARD：用户有任意 VIP 等级即可
   - 要求 PRO：用户必须是 PRO 或 MAX
   - 要求 MAX：用户必须是 MAX
4. 不满足则抛出 `NO_AUTH_ERROR` 并提示所需等级

#### 使用示例

```java
// 仅管理员可访问
@AuthCheck(mustRole = UserConstant.ADMIN_ROLE)
@PostMapping("/add")
public BaseResponse<Long> addSpace(...) { }

// 需要专业版及以上 VIP
@VipLevelCheck(mustLevel = "pro")
@PostMapping("/ai/generate")
public BaseResponse<PictureVO> aiGenerate(...) { }
```

### 论文描述建议

> 权限控制是 Web 应用安全的核心组成部分。传统 RBAC（基于角色的访问控制）模型通过用户-角色-权限三层结构实现细粒度授权，但同时也带来了数据库设计复杂、查询开销增加等问题。本项目采用简化方案：角色信息直接存储于用户表，通过自定义注解与 AOP 切面实现声明式权限校验。
>
> **数据模型设计。** 用户表包含 `userRole` 和 `vipLevel` 两个权限相关字段。`userRole` 取值为 `user`（普通用户）、`admin`（管理员）或 `vip`（VIP 用户）；`vipLevel` 取值为 `standard`（标准版）、`pro`（专业版）或 `max`（旗舰版）。这种设计省去了角色表、权限表及关联中间表，降低了数据库复杂度，但牺牲了权限配置的灵活性。
>
> **注解定义。** 项目定义了两个自定义注解。`@AuthCheck(mustRole)` 用于角色校验，标注在 Controller 方法上指定所需的角色；`@VipLevelCheck(mustLevel)` 用于 VIP 等级校验，限制接口仅对特定等级以上的 VIP 用户开放。两个注解均使用 `@Retention(RetentionPolicy.RUNTIME)` 保证运行时可反射获取。
>
> **切面实现。** `AuthInterceptor` 类使用 `@Aspect` 和 `@Around` 注解，拦截所有带有 `@AuthCheck` 注解的方法。切面通过 `RequestContextHolder` 获取当前 HTTP 请求上下文，调用 `UserService.getLoginUser()` 从 Session 中提取登录用户信息，然后比较用户的 `userRole` 字段与注解中指定的 `mustRole` 值。若角色匹配或用户为管理员，则调用 `joinPoint.proceed()` 放行请求；否则抛出 `BusinessException`，错误码为 `NO_AUTH_ERROR`。
>
> `VipLevelInterceptor` 的实现逻辑类似，但增加了层级比较。管理员角色自动跳过 VIP 等级校验；普通用户则根据枚举定义的等级层次进行判断：要求专业版时，用户必须是专业版或旗舰版；要求旗舰版时，用户必须恰好是旗舰版。校验失败时返回的错误消息包含所需等级的中文描述，便于用户理解。
>
> **设计权衡。** 该方案的优点在于实现简洁、性能高效——权限校验只需一次字符串比较，无需数据库查询或缓存同步。注解式的声明方法使权限需求与业务代码紧邻，便于理解和维护。然而，缺点也较为明显：权限与角色强绑定，无法实现资源级细粒度控制（如"只能操作自己创建的图片"）；新增角色或权限需要修改枚举代码并重新部署，不支持运行时动态配置。对于功能相对固定、角色划分简单的应用场景，这种简化方案是合理的选择。

## 7. 批量抓图上传

### 原文与实际对比

| 原文描述 | 实际情况 |
|---------|---------|
| 线程池并发处理 | ❌ 简单 for 循环，顺序执行 |
| CompletableFuture 异步编排 | ❌ 没有 |
| "3.8 倍效率提升" | ❌ 虚构数据 |
| MD5 去重机制 | ❌ 只有文件名去重（追加时间戳） |
| 批量入库优化 | ❌ 逐条调用 uploadPicture |

### 实际实现

#### 核心代码（PictureServiceImpl.uploadPictureByBatche）

```java
public Integer uploadPictureByBatche(PictureUploadByBatchRequest pictureUploadByBatchRequest, HttpServletRequest request) {
    // 1. 构造搜索 URL
    String fetchUrl = String.format("https://www.bing.com/images/async?q=%s&mmasync=1", searchText);

    Document document = Jsoup.connect(fetchUrl).get();
    Elements imageElements = document.select(".iusc");

    // 3. 简单 for 循环，顺序处理每张图片
    for (Element imageElement : imageElementsList) {
        // 3.1 解析图片 URL
        String fileUrl = m.get("murl");

        // 3.2 文件名去重（追加时间戳，非 MD5）
        String uploadName = prefix + SPLIT + System.currentTimeMillis();

        // 3.3 逐条调用上传方法
        PictureVO pictureVO = this.uploadPicture(fileUrl, pictureUploadByBatchRequest, loginUser);

        // 3.4 累计成功计数
        if (pictureVO != null) {

        }
    }
    return uploadCount;

```

#### 处理流程

```
Bing 图片搜索 URL
       ↓
Jsoup.connect().get()  ← 同步 HTTP 请求
       ↓
document.select(".iusc")  ← CSS 选择器提取图片元素
       ↓
for (Element element : elements) {  ← 顺序循环，非并发
       ↓
    解析 murl 属性
       ↓
    uploadPicture()  ← 单次上传（含 COS 上传、压缩、入库）
       ↓
    成功计数++
}
       ↓
返回成功数量
```

### 论文描述建议

> 批量抓图功能用于管理员快速导入网络图片至图床。功能基于 Jsoup HTML 解析库实现，支持从 Bing 和 360 图片搜索结果中批量抓取图片。
>
> **抓取流程。** 管理员输入搜索关键词和目标空间 ID，后端构造 Bing 图片搜索的异步 API URL，通过 Jsoup 发送 HTTP GET 请求获取 HTML 响应。使用 CSS 选择器 `.iusc` 定位图片元素，从 `murl` 属性中提取原图 URL。360 图片源的实现类似，通过 `img` 标签的 `src` 属性获取图片地址。
>
> **处理模式。** 抓取结果采用**同步顺序处理**：遍历图片元素列表，对每张图片依次执行下载、上传至 COS、生成缩略图、写入数据库。处理完一张后才处理下一张。代码中使用简单的 `for` 循环实现，没有使用线程池或 CompletableFuture 等并发机制。
>
> **文件名处理。** 为避免重名冲突，上传时在原文件名后追加当前时间戳（`prefix_1703123456789`），确保每张图片的存储路径唯一。项目未实现基于 MD5 或图片内容哈希的去重机制，相同图片可能被重复上传。
>
> **设计权衡。** 同步顺序处理的优点是实现简单、代码可读性高、便于调试和错误追踪。缺点是处理速度受限于网络延迟和图片大小，批量任务耗时较长。对于管理后台的低频操作场景，这种实现是可接受的折中方案。若需提升性能，可考虑引入线程池并发处理，但需要额外处理并发安全和错误隔离。

## 8. 以图搜图功能

### 原文与实际对比

| 原文描述 | 实际情况 |
|---------|---------|
| "身份认证密钥管理、请求签名生成" | ❌ 360 搜图是公开接口，无需认证和签名 |
| "相似度评分" | ❌ SoImageSearchResult 没有此字段 |
| "来源网站等信息" | ❌ 没有 source 字段 |
| "过滤无效或不可访问的链接" | ❌ 没有过滤逻辑，直接返回全部结果 |
| "2秒内完成" | ❌ 虚构数据，没有性能统计 |
| "云端安全图像检索...实践经验" | ❌ 虚构学术引用 |
| 门面设计模式（Facade Pattern） | ✅ 正确，SoImageSearchApiFacade |
| 提取图片 ID 获取 URL | ✅ 正确 |

### 实际实现

#### 门面类（SoImageSearchApiFacade）

```java
public class SoImageSearchApiFacade {
    public static List<SoImageSearchResult> searchImage(String imageUrl, Integer start) {

        String soImageUrl = GetSoImageUrlApi.getSoImageUrl(imageUrl);

        List<SoImageSearchResult> imageList = GetSoImageListApi.getImageList(soImageUrl, start);
        return imageList;
    }
}
```

**步骤1：GetSoImageUrlApi** - Jsoup 抓取页面提取库中 URL

```java
    // 从CSS选择器中提取背景图片URL
    Element imgElement = document.selectFirst(".img_img");
```

```java
    formData.put("img_url", imageUrl);
    // HuTool发送POST请求
    HttpResponse response = HttpRequest.post(url).form(formData).timeout(5000).execute();
    // 解析JSON响应
    JSONObject body = JSONUtil.parseObj(response.body());
    List<SoImageSearchResult> result = data.getBeanList("result", SoImageSearchResult.class);
    // 拼接完整图片URL（API返回的是分离的域名和key）
    for (SoImageSearchResult item : result) {
        String prefix = https ? "https://" + https + "/" : "http://" + http + "/";
        item.setImgUrl(prefix + item.getImgkey());
    }
    return result;
}
```

### 论文描述建议

> 以图搜图功能允许用户在私人空间中选择一张图片，查找网络上与之相似的图片资源。功能基于 360 搜图服务实现，采用门面设计模式（Facade Pattern）封装 API 调用细节。
>
> **调用流程。** 用户选择空间内的图片后，后端通过图片 ID 从数据库获取其在 COS 中的访问 URL，随后调用 `SoImageSearchApiFacade.searchImage()` 方法。该方法内部包含两个步骤：首先通过 Jsoup 抓取 360 搜图入口页面，从 HTML 响应中解析出图片在 360 图片库中的内部 URL；然后使用该内部 URL 调用 360 的相似图片推荐接口，获取 JSON 格式的搜索结果。
>
> **接口封装。** 360 搜图是公开服务，无需身份认证或请求签名。`GetSoImageUrlApi` 类使用 Jsoup 发送 GET 请求，通过 CSS 选择器 `.img_img` 定位图片元素，从 `style` 属性的 `background-image:url()` 中提取库内 URL。`GetSoImageListApi` 类使用 HuTool 的 `HttpRequest` 发送 POST 请求，响应为 JSON 格式。API 返回的图片信息是分离的（域名与 key），后端需要手动拼接完整的图片 URL。
>
> **数据模型。** 搜索结果 `SoImageSearchResult` 包含 `imgUrl`（完整图片地址）、`title`（标题）等字段。API 未提供相似度评分或来源网站信息，后端也未实现结果过滤逻辑，直接将全部结果返回给前端。Controller 层通过分页循环获取结果，直到凑够 50 条或无更多数据。
>
> **设计权衡。** 门面模式将两步 API 调用封装为单一入口，简化了 Controller 的调用复杂度，也为未来替换搜索引擎提供了扩展点。该实现的局限在于依赖 HTML 页面结构解析（第一步），若 360 搜图页面改版可能需要调整选择器；同时没有实现结果去重和质量过滤，可能返回重复或失效链接。
